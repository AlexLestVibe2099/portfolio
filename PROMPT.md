# Внутренний промт для создания сайта-портфолио

Создать отдельный сайт-портфолио на React + Vite для будущих веб-проектов. Сайт должен выглядеть как основной профессиональный ресурс, а не как очередной проект внутри портфолио. Не использовать имя автора как главный герой-бренд и не делать акцент на личной фотографии. Основной бренд сайта: `Webstack Lab`.

## Цель

Показать проекты как реальные продуктовые решения под понятную задачу. Не писать, что проект учебный. Первый кейс - `Telegram-заявочник`, система для бизнеса, которая собирает заявки с лендинга, уточняет детали через Telegram-сценарий и передает менеджеру готовую карточку заявки.

## Визуальный стиль

Сделать стиль отличным от солнечного лендинга проекта:

- dark premium-tech;
- графитовый/черный фон;
- крупная editorial-типографика;
- теплые off-white блоки как акцент;
- cyan и solar-yellow только как точечные акценты;
- визуально похоже на case-study/creative agency презентацию, но без копирования конкретных сайтов;
- без фото автора, вместо этого использовать абстрактный 3D/UI hero visual.

## Структура главной страницы

1. Header:
   - логотип `Webstack Lab`;
   - навигация: Кейсы, Подход, Контакт;
   - кнопка `Связаться`.

2. Hero:
   - короткий сильный заголовок о проектах, которые превращают идею в рабочий веб-продукт;
   - пояснение, что это портфолио сайтов, лендингов и Telegram-сценариев;
   - CTA `Смотреть кейсы`;
   - вторичная кнопка `Обсудить проект`;
   - справа/снизу визуал с UI-мокапами.

3. Коротко о себе:
   - писать не биографию, а позиционирование;
   - акцент на том, что проекты собираются под задачу: заявка, объяснение продукта, первый контакт, демонстрация результата.

4. Список кейсов:
   - первый кейс `Telegram-заявочник`;
   - карточка должна показывать: для кого, какую проблему решает, что внутри, результат, ссылка на отдельную страницу кейса.

5. Подход:
   - 3-4 этапа: понять задачу, собрать структуру, сделать интерфейс, подготовить к запуску.

6. Финальный CTA:
   - коротко подтолкнуть к контакту;
   - кнопка `Обсудить проект`.

## Страница кейса

Для каждого проекта нужна отдельная страница. В первом релизе сделать одну страницу `/case/telegram-zayavchnik` через hash-route.

Структура кейса:

1. Название проекта.
2. Для кого сделан.
3. Какую проблему решает.
4. Как было до.
5. Что сделал.
6. Что получилось.
7. Скриншоты/визуальные макеты.
8. Видео-демо как аккуратный placeholder-блок, чтобы позже заменить реальным видео.
9. Ссылка на продукт.

## Тон текста

Писать для потенциального клиента:

- без слов `учебный`, `просто практика`, `тренировочный`;
- объяснять выгоды и результат;
- формулировать просто, конкретно и уверенно;
- не перегружать техническими деталями.

## Технические требования

- React + Vite;
- одна кодовая база;
- адаптивная верстка;
- hash-routing без дополнительных роутер-зависимостей;
- готовность к деплою на Vercel;
- build должен проходить без ошибок.
